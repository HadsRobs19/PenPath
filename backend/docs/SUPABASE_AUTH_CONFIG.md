# Supabase Auth Configuration Guide
## MS6-6: Setup Supabase Auth for Parents & Teachers

---

## Overview

**WHO:** Supabase Auth Admin

**WHAT:** Configure email/password authentication for parent and teacher accounts

**WHERE:** Supabase Dashboard → Authentication Settings

**WHEN:** Before deploying parent login features

**WHY:** Parents need standard email/password; Supabase Auth handles security

---

## Step 1: Access Supabase Auth Settings

### In Supabase Dashboard:

1. Go to https://app.supabase.com
2. Select your **PenPath** project
3. Click **Authentication** in the left sidebar
4. Click **Providers**

---

## Step 2: Enable Email/Password Provider

### Configuration:

```
1. Click on "Email" provider
2. Toggle: "Email/Password" → ON ✅
3. Configure settings:
   ├─ Confirm email: YES (require verification)
   ├─ Double confirm changes: NO
   ├─ Secure password: YES (enforce strong passwords)
   └─ Autoconfirm: NO (require email verification)
4. Click "Save"
```

### Email Templates:

Supabase provides default templates. Customize if needed:

**Confirm signup email:**
- Template: `{{ confirmation_url }}`
- Subject: "Confirm your PenPath account"

**Password reset email:**
- Template: `{{ reset_url }}`
- Subject: "Reset your PenPath password"

## Step 3: Configure Password Requirements

### Go to: Authentication → Policies

```
Password Strength (RECOMMENDED CONFIGURATION):
├─ Minimum password length: 8 characters ✅
├─ Require special characters: YES (enforced) ✅
├─ Require numbers: Optional (recommended but not enforced)
└─ Allow simple passwords: NO ✅

RATIONALE:
├─ 8 characters: Good balance (not too restrictive)
├─ Special character: Prevents dictionary attacks
├─ Numbers: Most people use these anyway
├─ Industry standard: Similar to most banking apps
└─ Security: Adequate for educational app with kids' data
```

## Backend Implementation

### Environment Variables

In your `.env` files:

```bash
# .env.local
SUPABASE_URL=https://your-project.supabase.co
SUPABASE_ANON_KEY=eyJ...
SUPABASE_SERVICE_KEY=eyJ...
JWT_SECRET=your-secret-key-from-supabase

# Email
SMTP_HOST=smtp.sendgrid.net
SMTP_PORT=587
SMTP_USER=apikey
SMTP_PASS=SG.xxx...
SMTP_FROM=noreply@penpath.com
```

### Verify JWT in Backend

```javascript
// middleware/verifyToken.js
const jwt = require('jsonwebtoken');

function verifyToken(req, res, next) {
  const authHeader = req.headers.authorization;

  if (!authHeader) {
    return res.status(401).json({ error: 'No token provided' });
  }

  const token = authHeader.replace('Bearer ', '');

  try {
    const decoded = jwt.verify(
      token,
      process.env.JWT_SECRET
    );
    req.user = decoded; // Add to request
    next();
  } catch (error) {
    return res.status(401).json({ error: 'Invalid token' });
  }
}

module.exports = verifyToken;
```

### Use in Routes

```javascript
const verifyToken = require('./middleware/verifyToken');

// Protected route
app.get('/api/parent-profile', verifyToken, async (req, res) => {
  // req.user now contains decoded JWT
  // { sub: 'user-id', email: 'parent@email.com', ... }

  const userId = req.user.sub;
  // ... fetch user data
});
```

## Troubleshooting

### Issue: "Email already exists"
```
Solution:
├─ User already signed up
├─ Use /auth/parent-login instead
├─ Or use password reset flow
```

### Issue: "Invalid email verification link"
```
Solution:
├─ Link expired (usually 24 hours)
├─ Resend verification email
├─ Create new account
```

### Issue: "SMTP configuration failed"
```
Solution:
├─ Check SMTP credentials
├─ Verify port (587 for TLS, 465 for SSL)
├─ Check firewall allows outbound
├─ Use test email first
```

### Issue: "JWT verification fails"
```
Solution:
├─ Check JWT_SECRET matches Supabase
├─ Verify token hasn't expired (check exp claim)
├─ Check token format: "Bearer xxx"
├─ Decode at jwt.io to inspect
```

### Issue: "Redirect URL not allowed"
```
Solution:
├─ Add URL to Authentication → URL Configuration
├─ Include protocol (http://, https://)
├─ For mobile: penpath://auth/callback
├─ Restart app after adding
```

## Security Best Practices

### DO ✅
- ✅ **Use HTTPS only** in production
- ✅ **Never expose service key** to frontend
- ✅ **Verify all JWT tokens** on backend
- ✅ **Hash sensitive data** before storing
- ✅ **Use strong password policy**
- ✅ **Require email verification**
- ✅ **Rate limit auth endpoints**
- ✅ **Log all auth attempts**
- ✅ **Monitor failed logins**
- ✅ **Keep Supabase SDK updated**

### DON'T ❌
- ❌ **Store JWT in plain localStorage** (use secure cookies)
- ❌ **Send password in logs**
- ❌ **Accept passwords via URL**
- ❌ **Skip email verification**
- ❌ **Use weak JWT secret**
- ❌ **Expose Supabase URL in frontend**
- ❌ **Trust client-side password validation**
- ❌ **Allow unlimited login attempts**
- ❌ **Share JWT between users**
- ❌ **Store JWT longer than needed**

## Flow Diagrams

### Parent Signup Flow

```
Frontend (React)
├─ Form: email, password, name
├─ POST /auth/parent-signup
│
Backend
├─ Validate input
├─ Call Supabase Auth signup
├─ Create profile in database
├─ Send verification email
└─ Return user_id

Supabase
├─ Hash password
├─ Store in auth.users
├─ Generate confirmation token
├─ Send email

Parent's Email
├─ Receives: "Confirm your account"
├─ Clicks confirmation link
├─ Supabase verifies
└─ Account activated ✅
```

### Parent Login Flow

```
Frontend (React)
├─ Form: email, password
├─ POST /auth/parent-login
│
Backend
├─ Validate input
├─ Call Supabase Auth login
├─ Verify credentials
├─ Return JWT token
└─ Frontend stores JWT

Subsequent Requests
├─ Include: Authorization: Bearer {JWT}
├─ Backend verifies JWT
├─ RLS applies with auth.uid()
├─ Data filtered by user
└─ Return user-specific data ✅
```
